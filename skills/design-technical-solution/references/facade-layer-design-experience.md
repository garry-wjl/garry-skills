# 门面层设计经验

门面层用于承载跨模块通用基础类型与通用契约，例如领域基类、事件契约、统一返回对象、分页对象、通用请求上下文等。

## 强规约

1. **无变更也要说明**：如果本次不涉及门面层，技术方案中必须写明「本次无门面层变更」。
2. **只放通用契约**：门面层只承载跨层、跨模块复用的通用类型，不放具体业务领域逻辑。
3. **避免业务污染**：具体业务常量、业务状态、业务 DTO 不应放在门面层；应放在对应 领域层或客户端层模块。
4. **事件基础契约统一**：领域事件的基础结构（如事件 ID、类型、发送者、数据载荷）应统一设计，不允许每个领域重复定义一套事件基类。
5. **保持稳定性**：门面层是被多层依赖的基础模块，字段、方法、泛型变更要谨慎说明兼容影响。
6. **Spring Bean 注入必须使用 `@Resource`**：如涉及门面层接口的 Spring 实现类（例如基础设施层实现 `DomainEventPublisher`），注入 Spring Bean 时必须使用 `@Resource`；禁止使用 `@Autowired`、构造器注入、Setter 注入或通过 `ApplicationContext` 手动获取 Bean。

## 必选内容

### 1. 通用类型清单

| 类型 | 包路径 | 职责 | 关键字段/方法 | 新增/修改 | 说明 |
|------|--------|------|---------------|-----------|------|
| 领域实体基类 | `xxx.facade.domain` | 领域对象基类 | save/delete/domainValidate | 修改 | 增加审计约定 |
| 领域事件 DTO | `xxx.facade.event` | 领域事件载荷 | id/type/data/sender | 新增 | 统一事件格式 |
| 统一返回结果 | `xxx.facade.result` | 统一返回 | code/msg/data | 无变更 | 复用现有 |

### 2. 兼容性说明

如果修改已有门面层类型，必须说明：

- 是否影响客户端层 / 领域层 / 基础设施层 / 应用层 / 适配层的编译
- 是否需要迁移已有字段或方法
- 是否有默认值或兼容构造方式

### 3. 与其他层关系

- 领域层可依赖门面层中的领域基类、事件发布契约。
- 客户端层可复用通用统一返回结果 / 分页返回结果 等基础返回结构。
- 基础设施层可实现 门面层中的事件发布接口。

## 常见设计经验

### 经验 1：先判断是否真的通用

如果一个类型只服务某个业务领域，不要放在门面层。

### 经验 2：门面层变更要少而稳定

门面层被多个模块依赖，频繁变更会造成连锁影响。技术方案中应尽量复用已有通用类型。

### 经验 3：事件契约适合放在门面层

事件基础结构适合作为跨领域通用契约放在门面层；具体事件类型常量和领域事件语义应放在领域层。

## 自检清单

- [ ] 是否说明本次是否有门面层变更？
- [ ] 新增类型是否确实跨模块通用？
- [ ] 是否避免将业务领域逻辑放入门面层？
- [ ] 修改已有门面层类型时是否说明兼容影响？
- [ ] 门面层设计是否能对应 impl-facade-module 的实现项？
- [ ] 如涉及 Spring Bean 注入，是否统一使用 `@Resource`，且未使用 `@Autowired`、构造器注入、Setter 注入或 `ApplicationContext` 手动获取？
