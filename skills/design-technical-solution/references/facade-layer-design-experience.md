# Facade 层设计经验

Facade 层用于承载跨模块通用基础类型与通用契约，例如领域基类、事件契约、统一返回对象、分页对象、通用请求上下文等。

## 强规约

1. **无变更也要说明**：如果本次不涉及 Facade 层，技术方案中必须写明「本次无 Facade 层变更」。
2. **只放通用契约**：Facade 只承载跨层、跨模块复用的通用类型，不放具体业务领域逻辑。
3. **避免业务污染**：具体业务常量、业务状态、业务 DTO 不应放在 Facade；应放在对应 domain/client 模块。
4. **事件基础契约统一**：领域事件的基础结构（如事件 ID、类型、发送者、数据载荷）应统一设计，不允许每个领域重复定义一套事件基类。
5. **保持稳定性**：Facade 是被多层依赖的基础模块，字段、方法、泛型变更要谨慎说明兼容影响。

## 必选内容

### 1. 通用类型清单

| 类型 | 包路径 | 职责 | 关键字段/方法 | 新增/修改 | 说明 |
|------|--------|------|---------------|-----------|------|
| DomainEntity | `xxx.facade.domain` | 领域对象基类 | save/delete/domainValidate | 修改 | 增加审计约定 |
| DomainEventDTO | `xxx.facade.event` | 领域事件载荷 | id/type/data/sender | 新增 | 统一事件格式 |
| Result | `xxx.facade.result` | 统一返回 | code/msg/data | 无变更 | 复用现有 |

### 2. 兼容性说明

如果修改已有 Facade 类型，必须说明：

- 是否影响 client / domain / infra / application / adapter 编译
- 是否需要迁移已有字段或方法
- 是否有默认值或兼容构造方式

### 3. 与其他层关系

- domain 可依赖 Facade 中的领域基类、事件发布契约。
- client 可复用通用 Result / PageResult 等基础返回结构。
- infra 可实现 Facade 中的事件发布接口。

## 常见设计经验

### 经验 1：先判断是否真的通用

如果一个类型只服务某个业务领域，不要放 Facade。

### 经验 2：Facade 变更要少而稳定

Facade 被多个模块依赖，频繁变更会造成连锁影响。技术方案中应尽量复用已有通用类型。

### 经验 3：事件契约适合放 Facade

事件基础结构适合作为跨领域通用契约放 Facade；具体事件类型常量和领域事件语义应放 domain。
