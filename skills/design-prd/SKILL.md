# 编写 PRD 文档（产品需求文档）

**分类**：设计类

**产出用途**：本 skill 产出的 PRD 将作为 **design-technical-solution**（技术方案设计）技能的输入，用于编写技术方案文档并最终落码。PRD 中应包含足够清晰的业务目标、范围、功能点与优先级，便于技术方案设计时做「设计前 10 问」与模块拆解。

## When to Use

- 用户要求「写 PRD」、「写需求文档」、「出产品需求」、「写功能说明」或「先写 PRD 再做技术方案」。
- 在进入技术方案设计或代码实现前，需要先产出结构化的产品需求文档。

## 输入与输出

- **输入**：用户的业务想法、功能描述、口头/零散需求，或已有草稿。
- **输出**：一份结构化的 PRD 文档（Markdown），包含必选章节、**系统线框图、业务流程图、用例图**及必要的详细原型图，便于后续 **design-technical-solution** 据此提问并产出技术方案。

## 产出文档存放与命名

- **存放位置**：产出的 PRD 文档必须放在**项目根目录**下的 **`doc/产品方案/`** 文件夹中。若该路径不存在，则先创建 `doc` 与 `doc/产品方案` 目录再写入。
- **命名规则**：文件名须带**日期前缀**，格式为 `YYYY-MM-DD_<产品或需求名称>-PRD.md`。例如：`2025-03-16_会话关闭功能-PRD.md`。日期为文档编写或定稿日期。

## PRD 文档结构（必选章节）

**图示优先**：PRD 撰写时须**先产出系统线框图、业务流程图与用例图**，再展开文字需求；图示用于统一对产品形态与流程的理解，便于后续功能需求与原型图与之对齐。

1. **产品/需求背景**  
   简要说明为什么要做、解决什么问题、面向谁。

2. **目标与范围**  
   - **目标**：本需求要达成的业务目标（一句话或几条）。
   - **范围**：本次包含的功能/模块；**不做什么**（明确排除项）。

3. **系统线框图（必选，优先产出）**  
   在展开功能细节前，须先给出**系统级线框图**，表达：系统包含哪些主要模块/页面、模块间关系、主导航与信息架构。涵盖主要页面骨架（如首页、列表、详情、创建/编辑等）及关键元素位置，可区分 Web/移动端。形式：线框图、页面结构草图或 Mermaid/ASCII 结构图；嵌入文档并辅以简要说明。详见 [reference.md](reference.md) 中「线框图」小节。

4. **业务流程图（必选）**  
   须包含**业务流程图**，描述核心业务从触发到结束的步骤、分支与角色。例如：录入流水 → 分类/审核 → 入账；或注册 → 登录 → 创建账本 → 记账。可选用 Mermaid flowchart/sequence、BPMN 风格图或流程图截图；图中须体现主流程、异常/分支与涉及角色。详见 [reference.md](reference.md) 中「业务流程图」小节。

5. **用例图（必选）**  
   须包含**用例图**，表达参与者（角色）与用例（功能）的关系，以及用例之间的包含/扩展关系。明确「谁能用哪些功能」，便于技术方案设计时做权限与模块边界划分。可选用 Mermaid、PlantUML 或图示工具产出，嵌入 PRD 并说明图例。详见 [reference.md](reference.md) 中「用例图」小节。

6. **用户与场景**  
   主要用户角色、典型使用场景或用户故事（谁在什么情况下要完成什么），可与用例图对应。

7. **功能需求**  
   按模块或功能点列出，建议标注**优先级**（P0/P1/P2 或 MVP/后续迭代），便于技术方案设计时拆解子需求与实现顺序；功能点应与线框图、业务流程图、用例图一致。

8. **原型图/界面说明（必选）**  
   在已有线框图与流程图的基础上，补充**关键页面的详细原型图**：主要页面布局、核心操作流程（如创建/编辑/列表）、关键状态与异常态（如空态、错误提示）。可与线框图复用或细化；若为多端需求，需区分 Web/移动端。原型图可嵌入文档（图片）或附链接，并在正文中引用说明。详见 [reference.md](reference.md) 中「原型图」小节。

9. **非功能需求（可选但建议有）**  
   性能、安全、兼容、审计、多端、错误提示等与「怎么做」相关的约束或期望。

10. **与现有功能的关系（可选）**  
    是全新功能还是对现有功能的扩展/改造；若改造，需兼容或迁移的要点。便于技术方案设计时结合现有代码拆解。

11. **验收标准或成功指标（可选）**  
    如何算完成、可验收的条款。

## 编写约定

- **图示先行**：撰写 PRD 时先产出**系统线框图、业务流程图、用例图**，再写功能需求与详细原型；图示与文字须一致。
- 表述清晰、可执行：功能点尽量可被技术方案拆解为「模块变更清单」与「实现顺序」。
- **必须包含线框图、业务流程图、用例图及必要原型图**：线框图表达系统结构，业务流程图表达核心流程，用例图表达角色与功能关系，原型图表达关键页面与状态；便于技术方案设计时理解界面、流程与权限并拆解接口与数据。
- 优先级明确：多功能点时标明优先级，方便复杂 PRD 按子需求拆解。
- 不写技术实现细节：PRD 描述「做什么、为什么、给谁用」，技术选型与分层由 **design-technical-solution** 产出。

## 与下游技能衔接

- 本 skill 产出 PRD 后，使用者可将 PRD 交给 **design-technical-solution** 技能，由该技能进行「设计前至少 10 问」并产出技术方案；复杂 PRD 会被拆解为低耦合子需求后逐份产出技术方案，最终覆盖 PRD 全部内容。

## 使用完成后的最后一步：更新知识图谱

- **每次使用本技能完成交付后，最后一步须更新知识图谱**。根据本次产出（如新建/修改的 PRD 文档），创建或更新 ontology 中的实体与关系（如 Document、Project、Goal；`part_of`、`has_goal` 等），或调用 **ontology** 技能、或向 `memory/ontology/graph.jsonl` 追加操作记录，使图谱与项目当前状态一致。详见 [ontology/SKILL.md](../ontology/SKILL.md)。
